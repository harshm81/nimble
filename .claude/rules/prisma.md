# Prisma Rules

## Migration Workflow (Schema-first)

`schema.prisma` is the **single source of truth**. Never write raw SQL migrations by hand.

### My Role (Claude)
When a new table or field is needed, I update `prisma/schema.prisma` with the correct model/fields following all naming conventions.

### Your Role (Developer) ‚Äî run after I update the schema
```bash
npx prisma migrate dev --name <descriptive_name>
```
This one command: generates the SQL migration file, applies it to the DB, and regenerates Prisma Client.

### Production
```bash
npx prisma migrate deploy
```

## Never Do
- `npx prisma db pull` ‚Äî overwrites schema.prisma from DB, destroys `@map` / `@@map` conventions
- `npx prisma generate` manually ‚Äî `migrate dev` handles this
- Write raw SQL migration files by hand
- Direct `ALTER TABLE` / `CREATE TABLE` in the database
- Edit generated migration SQL files after creation
- Touch `_prisma_migrations` table manually

## Schema Rules

- Every model needs `@@map("snake_case_table_name")`
- Every field needs `@map("snake_case_column_name")`
- Model names: `PascalCase`
- Field names: `camelCase` in schema, `snake_case` in DB via `@map`
- Use `@db.DateTime(3)` on all `DateTime` fields for MySQL precision

## Required Columns ‚Äî Every Table

### Primary Key
Every table with a surrogate PK must use:
```prisma
id  BigInt  @id @default(autoincrement()) @db.UnsignedBigInt
```
- Type: `BIGINT UNSIGNED AUTO_INCREMENT` ‚Äî handles billions of rows without hitting limits
- Exception: tables with a natural unique key (e.g. `platform_tokens` uses `platform VARCHAR` as PK) ‚Äî surrogate `id` is omitted

### Timestamps
Every table must include these columns in this order at the end of the model:

```prisma
// For tables where the source platform exposes created/modified timestamps:
srcCreatedAt  DateTime  @map("src_created_at")  @db.DateTime(3)   // when source platform created the record
srcModifiedAt DateTime  @map("src_modified_at")  @db.DateTime(3)   // when source platform last modified ‚Äî incremental sync key

// For all tables:
createdAt     DateTime  @default(now()) @map("created_at")  @db.DateTime(3)   // when we inserted the row
modifiedAt    DateTime  @default(now()) @updatedAt @map("modified_at")  @db.DateTime(3)  // when we last upserted
```

**`src_*` field rules:**
- `src_created_at` / `src_modified_at` ‚Äî only include when the source API exposes these timestamps
- Analytics/reporting tables (GA4, Facebook Insights) have no source timestamps ‚Äî omit `src_*` columns
- Tables without a source created timestamp (e.g. inventory line items) omit `src_created_at` but keep `src_modified_at`

**Source platform field mapping:**
| Platform | src_created_at source | src_modified_at source |
|---|---|---|
| Cin7 | `CreatedDate` | `ModifiedDate` |
| Shopify | `created_at` | `updated_at` |
| Klaviyo | `created_at` / `created` | `updated_at` / `updated` |
| Facebook | `created_time` | `updated_time` |
| GA4 | ‚Äî (omit) | ‚Äî (omit) |

## Required Indexes ‚Äî Every Table

Every table must index `src_modified_at` (incremental sync key) and `synced_at`:

```prisma
@@index([srcModifiedAt], map: "idx_<table>_src_modified_at")
@@index([syncedAt],      map: "idx_<table>_synced_at")
```

## Analytics / Insight Tables ‚Äî Unique Constraint

Analytics and insight tables store daily aggregates (Facebook Insights, GA4 reports). These must have a `@@unique` on their natural key to prevent duplicate rows per date:

```prisma
// Facebook campaign insights ‚Äî one row per campaign per day
@@unique([facebookCampaignId, reportDate], name: "campaign_report_date")

// GA4 session data ‚Äî one row per date + dimensions
@@unique([reportDate, source, medium, campaign, deviceCategory], name: "session_dimensions")
```

Without this, a retry that re-fetches the same date range will insert duplicate rows.

## Naming ‚Äî Migration Files
- Format: `<timestamp>_<descriptive_name>` (auto-generated by Prisma)
- Good: `--name add_cin7_orders`
- Bad: `--name update`, `--name fix`, `--name changes`
